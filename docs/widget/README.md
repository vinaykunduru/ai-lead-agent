# Widget

## Purpose

The embeddable chat widget a company puts on their own website — the only surface a website visitor (an anonymous third party) ever interacts with directly. Everything else in this app exists to configure what this widget says and does.

## Installation

`/app/widget/[widgetId]/installation` shows a two-line snippet, generated by `generateInstallationSnippet()` (`src/modules/widget/installation-snippet.ts` — a pure function, directly unit-tested):

```html
<script src="https://<your-app-domain>/api/widget/sdk.js" data-widget-key="<public-key>" async></script>
<div id="ai-widget"></div>
```

The snippet exposes **only the public widget key** — never an organization id, database id, or any secret (`CLAUDE.md` §4). The `<script>` tag is the same byte-for-byte for every widget on the platform ([`GET /api/widget/sdk.js`](../api/README.md#get-apiwidgetsdkjs) — cached, `Cache-Control: public, max-age=300, stale-while-revalidate=3600`); all per-widget appearance and behavior is fetched at runtime from `/api/widget/config?key=...`.

## Configuration

| Screen | Controls |
|---|---|
| `/app/widget/[widgetId]` | Name, description, default language, status (draft/active/disabled/archived) |
| `/app/widget/[widgetId]/appearance` | Primary/accent color, launcher position/icon, border radius, color scheme, font, logo/avatar, width/height |
| `/app/widget/[widgetId]/behaviour` | Welcome message, suggested questions, typing indicator, branding visibility, offline message, timestamps, auto-open + delay |
| `/app/widget/[widgetId]/domains` | Allowed-domain list (see [Security](#security-model)) |
| `/app/widget/[widgetId]/keys` | Rotate the public key |
| `/app/widget/[widgetId]/preview` | Live preview of the configured widget |

Status changes go through a **separate permission** (`widget.publish`) from other field edits (`widget.update`) — see [API Reference](../api/README.md#widgets--company-side). A widget is never hard-deleted; "delete" archives it (`status: "archived"`), consistent with the app's general no-permanent-delete convention for real assets ([Database → RLS](../database/rls.md#policy-matrix-by-table-group)).

## Security model

Backend resolution chain for **every** widget request, per `CLAUDE.md` §4:

```
public widget key → widgets row → organization → ai_configuration + knowledge base → allowed actions
```

This resolution happens server-side, on every request — the public key is the only thing the client can ever supply.

### Three independent gates on public endpoints

1. **Public widget key** resolves to a widget/organization. Keys are managed like credentials: rotating (`POST /api/widgets/:widgetId/keys/rotate`) revokes the old key (kept for audit history, never deleted) and issues a new one — the rotate response is the **only** place the new key's full value is ever returned; there is no "reveal" endpoint afterward.
2. **Domain allowlist**: the caller's `Origin`/`Referer` host must match an enabled row in `widget_domains`, managed at `/app/widget/[widgetId]/domains`.
3. **Rate limiting** (`src/modules/widget/rate-limit.ts`): in-memory, per-server-instance sliding window, 60 requests/60s keyed by `x-forwarded-for`. Explicitly **not** a distributed limiter in Phase 1 (no Redis, per `CLAUDE.md` §2) — it throttles per instance only.

Every rejection reason other than rate-limiting collapses into one generic `400` — invalid key, wrong domain, and malformed input are all indistinguishable to the caller, by design, so an anonymous party can't learn which check failed. See [API Reference — Widget public endpoints](../api/README.md#widget--public-unauthenticated).

### What public endpoints never return

`resolvePublicWidgetConfig()` returns a public-safe DTO only: `{ name, defaultLanguage, appearance, behaviour }`. Never `organizationId`, `widgetId`, knowledge base contents, AI behaviour configuration, embeddings, or any secret.
