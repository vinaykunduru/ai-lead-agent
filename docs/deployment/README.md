# Deployment

Production stack: **Vercel** (Next.js hosting) + **Supabase** (Postgres, Auth, Storage) + **Inngest** (background jobs) + **Sentry** (error tracking). Deployed at `https://agent.bloomdigital.co.in`.

## Vercel

- Framework preset: Next.js. Build command: `pnpm build` (or Vercel's default, which resolves to the same via `package.json`). Turbopack is the default bundler for both `next dev` and `next build` in Next.js 16.
- Every server environment variable from [Getting Started → Environment variables](../getting-started/README.md#environment-variables) must be set in the Vercel project's Environment Variables panel — production, and separately for preview deployments if you want preview builds to hit a real (ideally staging) database.
- **Connection pooling is a real, previously-hit production constraint.** `src/db/client.ts` caps the `postgres()` client at `max: 1` per instance:
  > Each Vercel serverless invocation gets its own module instance of this client, and many can run concurrently. `postgres.js` defaults to `max: 10` per instance, which exhausted Supabase's pooler (`pool_size: 15`) under modest concurrent traffic (`EMAXCONNSESSION`). Capping at 1 per instance lets far more concurrent invocations fit under the pooler's total limit.

  If you see `EMAXCONNSESSION` errors in Vercel logs again, this is the first place to look — either traffic has grown past what the pooler's total limit supports at `max: 1`, or the pooler's own `pool_size` needs raising in Supabase, not the per-instance cap.
- `DATABASE_URL` in production must be the Supabase **Session pooler** URI, not "Direct connection" — see [Getting Started](../getting-started/README.md#session-pooler-vs-direct-connection).

## Supabase

- Apply migrations to the production project the same way as local (`pnpm db:migrate` with `DATABASE_URL` pointed at production) — there is no separate production migration tool.
- Authentication → URL Configuration → Redirect URLs must include `https://<production-domain>/auth/confirm`, or first-owner invite links fail.
- The service-role key used in production is the same one validated by `SUPABASE_SERVICE_ROLE_KEY` — treat it as a full-access credential to every table (RLS-bypassing), not a scoped API key.
- Configure custom SMTP in Supabase for anything beyond light testing — the built-in email sending works out of the box but at a very low rate limit (a handful of sends per hour before "email rate limit exceeded").

## Inngest

`INNGEST_EVENT_KEY` and `INNGEST_SIGNING_KEY` are required in production (optional locally, where the Inngest dev server handles this without them). Inngest discovers the app's registered functions via `/api/inngest` (`GET`/`POST`/`PUT`, auto-generated by the `inngest/next` `serve()` helper) — no separate deploy step is needed beyond the app itself being reachable; Inngest calls back into the deployed URL.

## Sentry

`NEXT_PUBLIC_SENTRY_DSN` (client-visible DSN) and `SENTRY_AUTH_TOKEN` (server-only, used at build time to upload source maps) are both optional — the app runs without them, just without error tracking/readable stack traces in Sentry.

## CI

`.github/workflows/ci.yml` runs on every push to `main` and every pull request: install → typecheck → lint → test → production build. The [cross-tenant isolation test suite](../testing/README.md#cross-tenant-isolation-tests) only runs for real if `TEST_DATABASE_URL`, `TEST_SUPABASE_SERVICE_ROLE_KEY`, `TEST_SUPABASE_URL`, `TEST_SUPABASE_ANON_KEY` repository secrets are configured (pointing at a disposable Supabase project) — otherwise it skips, same as running locally without those vars. CI does not deploy; Vercel's own GitHub integration handles deployment on push to `main` independently.

## Production checklist

Before shipping a change that touches tenant data, auth, or the widget:

- [ ] `pnpm typecheck && pnpm lint && pnpm test && pnpm build` all pass locally
- [ ] Any new tenant table has RLS enabled with a policy in the same migration ([Database — RLS](../database/rls.md#what-every-new-tenant-table-must-ship-with))
- [ ] Any new tenant-data module has a [cross-tenant isolation test](../testing/README.md#cross-tenant-isolation-tests)
- [ ] Migrations applied to the production Supabase project (`pnpm db:migrate` against production `DATABASE_URL`)
- [ ] New/changed env vars set in Vercel for both Production and Preview environments
- [ ] If UI/UX changed, verified against a real production build (`next build && next start`), not just `next dev` — this codebase has repeatedly hit bugs (RLS-adjacent, CSS collapse, ESM-only dependency crashes) that only manifest in a real production build or real production traffic, never in dev. See [Troubleshooting](../troubleshooting/README.md).
